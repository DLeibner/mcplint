# mcplint rule catalog

Every rule is a pure, deterministic function over the captured `tools/list` snapshot. No LLM calls, no network, no API keys. Thresholds are configurable in `.mcplintrc.json`; the defaults below are anchored in [Anthropic's tool-writing guidance](https://www.anthropic.com/engineering/writing-tools-for-agents) and the [MCP tools spec](https://modelcontextprotocol.io/specification/2025-06-18/server/tools).

Severity legend: **error** — actively harms tool selection; **warn** — measurable degradation; **info** — advisory, no score deduction.

---

## Tier 1 — hygiene

### descriptions/missing

**error** · A tool has no description. The model must guess intent from the name alone; the description is the single highest-leverage piece of context for tool selection.

### descriptions/too-short

**warn** · Description shorter than `minChars` (default 20). Carries no disambiguation signal — the model needs enough context to pick this tool over its neighbours without trial calls.

### descriptions/too-long

**warn** · Description longer than `maxChars` (default 1500). Long descriptions are paid for in every conversation. Detail that only matters once a parameter is used belongs in that parameter's description.

### schemas/param-desc-missing

**warn** · An input-schema property has no description. Undescribed parameters get filled by guesswork.

### surface/tool-budget

**warn/error** · Tool count above `warnAt` (default 20) warns; above `errorAt` (default 40) errors. Selection accuracy degrades as the surface grows: more near-neighbours to confuse, more tokens spent before the first user message.

### surface/token-budget

**warn** · The serialized `tools/list` payload exceeds `maxTokens` (default 10 000, counted with `o200k_base`, approximate). This payload is injected into every conversation — a per-message tax.

### naming/convention

**warn** · Tool names mix conventions (snake_case vs camelCase vs kebab-case). Flags the minority style. One convention makes unseen names predictable.

### annotations/missing-hints

**warn** · A tool with a mutating verb (`create_`, `delete_`, `cancel_`, …) declares neither `readOnlyHint` nor `destructiveHint`. These hints are machine-readable: clients use them to decide when to ask the user for confirmation. Prose like "confirm with the user first" is invisible to that machinery. Also notes (info) missing `title` on mutating tools.

### schemas/loose

**warn** · Bare `type: object` with no properties, untyped arrays (no `items`), or `additionalProperties: true`. Loose schemas turn validation off exactly where the model is most likely to hallucinate argument shapes. An empty object with `additionalProperties: false` (a no-input tool) is fine.

### schemas/complexity-budget

**warn** · Input schema deeper than `maxDepth` (default 3), wider than `maxProps` top-level properties (default 12), or using `oneOf`/`anyOf` more than `maxCombinators` times (default 2). Deep, wide, combinator-heavy inputs are a constraint puzzle solved before every call.

## Tier 2 — design

### design/overlap-cluster

**warn** · Clusters of tools that look like one tool wearing several names, detected two ways: (1) ≥ `minPrefixCluster` tools (default 3) sharing their first two name tokens; (2) pairwise description similarity via trigram Jaccard ≥ `similarity` (default 0.55). Near-synonym tools are the classic cause of wrong-tool calls. Consolidate with include-flags or a mode parameter.

### design/crud-mirror

**warn** · A get/create/update/delete family over the same noun — the fingerprint of an OpenAPI-autogenerated surface. It mirrors your REST API, not your users' intents. Agents work in workflows, not endpoints.

### design/confusable-params

**warn** · The same concept spelled differently across tools (`hotel_id` vs `hotelId` vs `hotel-id`). The model has to carry a per-tool spelling table, and arguments bleed across tools. One concept, one name, everywhere.

### design/enum-in-prose

**warn** · A parameter description enumerates values ("one of: a, b, c") but the schema declares no `enum`. Prose values are invisible to validation — misspellings surface as runtime errors instead of schema errors.

### design/duplicate-leading-words

**warn** · ≥ `minTools` (default 2) tools share their first `leadingWords` (default 15) description words. Models disambiguate primarily by leading tokens; identical openings bury the distinguishing signal and usually mean a copy-pasted block that belongs in one place. Reports the duplicated token cost.

### design/client-directives

**info → warn** · The description contains imperatives aimed at the client/agent — rendering instructions ("RENDER RESULTS IMMEDIATELY", "always display the map widget") or directives about *other* tools ("do not call X"). That is prompt injection into your own surface: paid for in every conversation and unevenly obeyed. Escalates to **warn** when the same block is duplicated across tools. Reports the token cost. Self-referential usage guidance ("do not call this speculatively") is *not* flagged — see the next rule.

### design/negative-guidance-present

**info (positive)** · The tool documents when *not* to use it ("Only call when the user explicitly asks…", "Do not call speculatively"). This is one of the highest-value lines a description can carry. Credited, never deducted — and worth preserving through any consolidation.

### design/list-no-limit

**warn** · The tool's name or description implies list/search output but the schema has no `limit`/pagination parameter. Result size is unbounded; the caller must be able to ask for less.

### design/enum-combination-unencoded

**info** · The same enum value-set is reused across multiple tools. If not every combination is valid (e.g. not every product supports every tier), the schema cannot express it and the model discovers the rules via runtime errors. Flagged for a validity review.
